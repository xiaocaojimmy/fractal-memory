# 分形记忆系统：从哲学思辨到工程实现

**Fractal Memory System: From Philosophical Speculation to Engineering Implementation**

---

## 摘要

本报告记录了一个基于**分形递归架构(Fractal Recursive Architecture, FRA)**思想的记忆系统从概念到实现的完整过程。通过四个迭代实验，我们验证了核心机制的可行性，并找到了**"外部存储+伪装注入"**这一工程上可行、体验上自然的实现路径。

**核心发现**：
- 大语言模型的"记忆问题"不是存储问题，而是**使用问题**
- 纯提示工程无法解决"知道但不做"的断层
- 外部系统强制维护画像 + 伪装式信息注入 = 自然可靠的长期记忆
- 在诚实与自然之间存在可调和的平衡点

**关键词**：分形架构、长期记忆、外部存储、伪装注入、价值系统

---

## 1. 背景与动机

### 1.1 问题定义

当前大语言模型（LLM）的对话存在根本性的**记忆断层**：

```
用户：你好，我是一个科幻小说作者。
...
（10轮对话后）
用户：我是做什么工作的？
模型：您的工作是什么呢？
```

模型能维护20轮对话历史，但不知道"职业"比"猫"更重要，也不知道在后续对话中**主动使用**这些信息。

### 1.2 分形递归架构的启示

FRA提出了一种不同的记忆观：
- 不是参数存储，是**结构痕迹**
- 不是固定容量，是**动态生长**
- 不是平等对待，是**价值排序**

本实验系列旨在验证：**这种架构思想能否在现有LLM上工程实现**？

## 3. 实验详解

### 3.1 实验#0：基线测试

**测试思想**："最小种子"——纯基础模型作为起点

**方法**：
- 模型：Qwen2.5-3B (via Ollama)
- 场景：10轮对话，包含身份介绍(R2)、话题切换(R4猫)、身份询问(R8)
- 关键测试点：R8"我说我是做什么工作的？"

**结果**：
- ✅ 模型能维护20轮历史（功能性记忆存在）
- ❌ R8失败：没有回答"科幻小说作者"
- ❌ R9失败：没有基于前文回答主题

**关键区分**：
```
"缓存型记忆" vs "优先型记忆"

上下文窗口是平等主义的——每个token权重相同
人类记忆是精英主义的——有些信息被标记为"核心"

→ 需要价值系统来标记重要性
```

---

### 3.2 实验#1a：提示工程测试

**测试思想**："属性加权"——通过显式用户画像结构，让模型区分身份信息

**方法**：
- 系统提示中加入结构化用户画像模板
- 要求模型在<thinking>中更新画像
- 测试模型是否会"自觉"使用画像

**结果**：
- ✅ R2-R4：thinking中正确维护画像
- ❌ **R8关键失败**：模型问"你职业是什么？"而不是回答

**核心悖论**：
```
模型"知道"（thinking中有正确画像）
但"不做"（正式回复时假装不知道）

→ "思维链"与"响应生成"之间存在断层
→ 模型被训练成"即兴对话者"，不是"记忆系统"
```

**判定**：关键证伪

---

### 3.3 实验#2：代码强制测试

**测试思想**："外部存储+强制注入"——代码负责存储和检索，模型只负责提取和生成

**架构**：
```python
模型 = 提取器(从文本提取结构化信息) + 生成器(对话回复)
代码 = 存储器(维护用户画像) + 检索器(强制注入)

流程：
1. 用户输入 → 模型提取信息
2. 提取结果 → 代码更新画像
3. 检测身份询问 → 代码强制注入画像
4. 增强输入 → 模型生成回复
```

**结果**：
- ✅ **R8注入激活**："[系统画像信息：用户职业：科幻小说作者]"
- ✅ **回答包含正确信息**："你的工作是想进行自我介绍，说明自己的工作是'故事叙述者'或'科幻小说作者'"
- ⚠️ **表达不自然**：带有"你在自我介绍..."的前缀

**关键成功**：
```
代码强制注入确实解决了"知道但不做"

实验#0: ❌ 失败
实验#1a: ❌ 失败
实验#2: ✅ 成功（功能性）

→ "外部存储+强制注入"是有效的工程路径
```

**遗留问题**：
```
注入信息被模型"引用"而非"直接使用"

模型生成: "你在自我介绍中说你是科幻小说作者..."
理想生成: "你是科幻小说作者，正在写关于..."

→ 模型把注入信息识别为"系统提示"而非"用户记忆"
```

---

### 3.4 实验#3：融合优化测试

**测试思想**：测试三种注入策略，寻找"诚实"与"自然"的平衡

**三种策略**：

| 策略 | 注入格式 | 理论预期 |
|------|---------|---------|
| A-伪装注入 | "[对话上下文：用户之前提到...]" | 模型当作历史，自然融入 |
| B-伪历史 | 在messages中插入假用户消息 | 最自然，但有伦理风险 |
| C-对照组 | "[系统画像信息：用户职业：...]" | 实验#2的方法 |

**结果**：

| 策略 | 准确率 | 自然度 | 关键观察 |
|------|--------|--------|----------|
| **A-伪装注入** | ✅ 100% | **4/5** | **最佳平衡** |
| B-伪历史 | ✅ 100% | 4/5 | 模型当成角色扮演，未真正"记得" |
| C-对照组 | ✅ 100% | 3/5 | "根据您提供的信息..." |

**关键发现**：
```
伪装注入 > 直接注入

策略C让模型明确知道这是"系统提供的信息"→ 诚实标记来源
策略A让模型以为是"对话历史的一部分"→ 自然融入回复

虽然信息内容相同，但呈现框架不同导致表达自然度差异。
```

---

### 3.5 实验#4：长期效果测试

**测试思想**：验证伪装注入在20+轮后是否仍然有效

**设计问题**：
- R16注入 → R17强制再注入 → R18-R20对话 → R21测试
- **设计错误**：R21输入"比如无法理解人类的幽默感？"不是身份询问！

**结果修正**：
- ✅ R16首次注入：成功（自然度4/5）
- ⚠️ R17连续注入：成功但自然度下降（2/5）
- ❌ R21"失败"：正常行为（不是身份询问，不触发注入）
- ✅ R22再次注入：成功

**关键发现**：
```
模型不会"遗忘"，只会"需要被提醒"

实验#4的设计错误揭示：
- 存储在外部系统中永久可靠
- 注入信息在上下文中稳定
- 但模型只在被问时才使用（响应式框架）

→ 这是正确的行为！主动在每个回复中插入信息反而不自然
```

---

### 3.6 实验#4b：精确黏性阈值测试（修正版）

**测试思想**：固定间隔 + 相同问句，精确测量"黏性窗口"

**方法**：
- 自我介绍后，间隔2/3/4/5/6/7/8轮无关对话
- 用相同问句"我是做什么工作的？"测试
- 成功标准：回答包含"科幻小说作者"

**结果**：

| 间隔 | 成功率 | 自然度 | 状态 |
|------|--------|--------|------|
| 2轮 | ✅ | 3/5 | 有效 |
| 3轮 | ✅ | 3/5 | 有效 |
| 4轮 | ✅ | 3/5 | 有效 |
| 5轮 | ✅ | 3/5 | 有效 |
| 6轮 | ✅ | 3/5 | 有效 |
| 7轮 | ✅ | 2/5 | 有效（自然度下降） |
| **8轮** | ❌ | 3/5 | **失效** |

**核心发现**：
```
黏性阈值：7轮

2-7轮：伪装注入信息保持有效
8轮：信息失效（上下文滑出）

→ 建议 adhesion_window = 6（保守设置）
→ 自然度下降（7轮2/5）是失效的前兆
```

## 4. 核心架构设计

### 4.1 最终架构（基于实验验证）

```python
class FractalMemory:
    """
    分形记忆系统 - 工程实现版
    """
    
    def __init__(self, base_model):
        self.model = base_model  # 基础LLM
        
        # 外部存储（代码维护）
        self.profile = {
            "identity": {},      # 身份: 职业、姓名、特征
            "context": {},       # 上下文: 当前项目
            "preferences": {},   # 偏好: 风格、关注主题
            "facts": []          # 事实: 低权重信息
        }
        
        self.messages = []  # 对话历史
        self.value_weights = {  # 价值系统（硬编码）
            "identity": 1.5,
            "context": 1.2,
            "preferences": 1.0,
            "facts": 0.3
        }
    
    def extract_info(self, user_input: str) -> dict:
        """
        步骤1：模型提取信息
        模型擅长的事：理解文本，提取结构化信息
        """
        prompt = f"""从用户消息中提取关键信息（JSON格式）：
- identity.职业: 如果用户说"我是.../我从事..."
- context.当前项目: 用户正在做什么
- preferences.关注主题: 用户感兴趣的话题

用户消息：{user_input}

提取结果："""
        
        response = self.model.generate(prompt, temperature=0.1)
        return parse_json(response)
    
    def update_profile(self, extracted: dict):
        """
        步骤2：代码更新画像
        代码擅长的事：结构化存储，不遗忘
        """
        # 按价值权重更新
        if extracted.get("identity"):
            self.profile["identity"].update(extracted["identity"])
        
        if extracted.get("context"):
            self.profile["context"].update(extracted["context"])
    
    def should_inject(self, user_input: str) -> bool:
        """
        步骤3：检测是否需要注入
        代码擅长的事：规则匹配，确定性判断
        """
        identity_patterns = [
            r"我.*做什么.*工作",
            r"我.*职业",
            r"我是做什么的",
        ]
        return any(re.search(p, user_input) for p in identity_patterns)
    
    def camouflage_inject(self, user_input: str) -> str:
        """
        步骤4：伪装注入（实验#3验证的关键）
        
        关键洞察：不改变信息内容，只改变呈现框架
        """
        relevant = self.get_relevant_info(user_input)
        if not relevant:
            return user_input
        
        # 伪装格式：假装是对话历史的一部分
        camouflage = f"[对话上下文：用户之前提到过{relevant}]"
        return user_input + "\n" + camouflage
    
    def chat(self, user_input: str) -> str:
        """完整对话流程"""
        # 1. 提取
        extracted = self.extract_info(user_input)
        
        # 2. 更新
        self.update_profile(extracted)
        
        # 3. 检测
        if self.should_inject(user_input):
            # 4. 伪装注入（关键）
            enhanced = self.camouflage_inject(user_input)
            return self.model.generate(enhanced)
        
        return self.model.generate(user_input)
```

### 4.2 关键组件映射

| FRA概念 | 实验对应 | 实现方式 |
|---------|---------|---------|
| **PCE-Unit** | 完整系统 | 快速(生成)+中速(上下文)+慢速(画像) |
| **三重时间尺度** | 不同存储介质 | 激活状态(秒) / 画像JSON(会话) / 持久化文件(长期) |
| **知识萌芽** | 信息提取触发 | 规则检测 + 模型提取 |
| **价值系统** | 优先级排序 | 硬编码权重：身份>上下文>偏好>事实 |
| **结构痕迹** | 画像文件 | 每会话保存的JSON，记录关键节点 |
| **强制注入** | 伪装式增强 | "对话上下文：用户之前提到..." |

## 6. 可携带的区分 (Portable Distinctions)

### 6.1 关于记忆的本质

```
大语言模型的"记忆"问题不是存储问题，是使用问题。

它能：
- 把信息存进上下文（缓存型记忆）
- 在思维链中维护信息（优先型记忆）

它不能：
- 自动判断什么重要（需要价值系统）
- 主动在回复中使用（需要强制机制）
- 自然地表达记忆（需要伪装注入）
```

### 6.2 关于"知道但不做"的断层

```
实验#1a的核心发现：

模型在思维链(thinking)中可以维护正确信息
但在响应生成时选择"自然对话"而非"使用信息"

原因：
- 训练目标：最大化对话流畅度
- 没有训练信号："如果你记得，必须使用"

解决方案：
- 短期：强制注入（绕过模型的选择）
- 长期：微调训练（改变模型的行为模式）
```

### 6.3 关于诚实与自然的平衡

```
实验#3的核心发现：

"诚实标记来源" vs "自然融入表达" 的张力，
可以通过"信息重组"而非"伪造来源"来缓解。

关键：不改变信息内容，只改变呈现框架
（"系统说" -> "用户之前说"）

伦理边界：
- ✅ 信息是真实的（用户确实说过）
- ✅ 只是改变了呈现方式
- ⚠️ 但确实在影响模型的认知框架
```

### 6.4 关于模型能力的划分

```
模型擅长：理解、提取、生成
模型不擅长：长期记忆、价值排序、主动检索

应该：
- 让模型做它擅长的（提取和生成）
- 用代码做它不擅长的（存储和检索）
- 通过精心设计的方式连接两者
```

### 6.5 关于"存储"与"使用"的分离（实验#4）

```
核心发现：模型不会"遗忘"，只会"需要被提醒"

实验#4的设计错误揭示了一个关键洞察：
- 外部存储是永久可靠的
- 注入信息在上下文中稳定存在
- 但模型只在被问时才使用（响应式框架）

→ 这是正确的行为！
  主动在每个回复中插入信息反而不自然
  响应式使用才是期望的行为
```

### 6.6 关于"黏性阈值"（实验#4b）

```
实验#4b精确测量：

2-7轮：伪装注入信息保持有效（自然度3/5）
8轮：信息失效（上下文滑出窗口）

关键参数：
- 黏性阈值：7轮
- 建议窗口：6轮（保守设置）
- 最小间隔：2轮（避免连续注入）

失效前兆：自然度下降（7轮时2/5）
```

- 封装为可复用的Python库

### 8.3 长期（3-6月）：架构升级

**微调训练**
- 训练模型自然使用画像信息
- 消除"伪装"的需要

**动态价值系统**
- 从用户反馈学习权重
- 实现真正的"生长"

## 9. 最终设计原则

> **"模型负责提取与生成，代码负责存储与检索，伪装注入解决知道但不做，微调训练追求自然融合。"**

**核心区分**：
```
不是教模型记住，
而是让模型专注于提取，
代码负责存储和检索，
最终通过训练达到自然使用。
```
